### bootc 기반 OS를 어떻게 구축/배포했나요?

### 구축

- 별도의 빌드 노드를 사용, ContainerFile 작성 후 컨테이너 이미지 빌드 및 Bootc Image Builder로써 ISO 파일로 이미지를 배포하여 bootc 기반 OS 구축

### 배포

- Proxmax ISO 저장소 내에 SCP로 파일을 전송하여 배포 진행

## 일반적인 OCI 컨테이너와 bootc 컨테이너가 어떤 차이가 있었나요?

**OCI 컨테이너** 

- runc를 사용, 리눅스 네임스페이스와 Cgroup을 기반으로 구현
- Host OS의 커널을 공유하는 방식으로 사용함

**bootc 컨테이너**

- 리눅스 커널 자체를 포함하고 배포함
- 개별 리눅스 커널을 가지고 있음


### 일반적인 OS와 bootc 기반 OS가 어떤 차이가 있나요? (bootc OS 부팅 후 실제 사용해보는 것 권장)

- 일반적인 OS의 경우 OS 자체를 이전 버전으로 부분적으로 롤백하는 것이 불가능함
- OS의 경우에는 패키지의 집합(apt)등으로 관리해야 되지만, bootc의 경우 미리 정의된 동작으로 관리가 가능하여, 버전 컨트롤이 가능함 ⇒ bootc 기반에서는 라이프사이클 바인딩이 가능함
- bootc 기반 OS에서는 코드 및 커밋 단위로써 OS관리가 가능함

### 다른 솔루션 대비 bootc 솔루션은 무엇에 장점이 있을까요?

### **vs Packer**

- Packer의 경우 AWS의 인스턴스 템플릿과 같이 AMI 생성을 자동화 해주는 도구
- OS 내에 패키지 혹은 환경설정을 작성하여 선언형(?)으로 OS 이미지 빌드 자동화가 가능함
- Packer의 경우 이미지 롤백 시에 사용자 데이터가 증발함
- bootc의 경우 컨테이너 이미지를 통해서 이미지 별 캐시가 가능하여, 업그레이드시 속도가 빠르며 **롤백 시에 OS이미지 교체가 아닌 이미지 레이어만 변경하여, 사용자 데이터 보존이 가능함**
